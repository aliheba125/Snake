# التحليل الديناميكي (Dynamic / Emulation Analysis) — libengine.so

> جولة تحليل ديناميكي فعلية عبر **محاكاة المعالج (Qiling 1.4.6 / Unicorn 2.1.4)** لتنفيذ كود المكتبة الأصلية ورصد سلوكها الحقيقي وقت التشغيل — لا استنتاج.

---

## 0. تدرّج الثقة

صُنّفت كل نتيجة حسب قوة الدليل:

| الرمز | المعنى |
|-------|--------|
| 🟢 **مُثبَت** | مرصود فعليًا بالتنفيذ/المحاكاة + مؤكَّد من مصدرين مستقلين |
| 🟡 **دليل قوي** | مدعوم بأدلة متماسكة لكن يحتمل تفسيرًا بديلًا |
| 🔴 **غير موثوق** | نتيجة أداة فشلت / أثر خادع — لا تُعتمد |

## 1. بيئة التحليل وقيودها

- الصندوق الرملي **لا يملك KVM ولا تسريعًا عتاديًا** (`/dev/kvm` غير موجود، لا `vmx/svm`).
  → **محاكي أندرويد كامل (بواجهة + Frida) غير عملي هنا**؛ سيكون بطيئًا لدرجة التعذّر.
- البديل المُستخدَم (وهو أعمق على مستوى الكود الأصلي): **محاكاة المعالج ARM64 عبر Qiling/Unicorn** مع نظام ملفات bionic أندرويد (`arm64_android`).
- **الحدّ الموضوعي:** المحاكاة تُنفّذ الكود خارج بيئة أندرويد الحقيقية (لا JavaVM/JNIEnv حقيقي، لا Zygote، لا نظام ملفات الجهاز)، لذلك تتوقّف الدوال المُعتَّمة بعد عشرات التعليمات عند اعتمادها على حالة غير متوفّرة. ما رُصد قبل التوقّف **حقيقي**؛ وما بعده يحتاج جهازًا فعليًا.

## 2. 🔴 لماذا فشل التعداد الساكن لنداءات النظام؟ (تصحيح للسجل)

المحاولة الأولى (`syscall_enum.py`، فكّ خطي + تتبّع `x8`) أعطت **25,804 موضع `svc`** لكنها انحصرت في قيمتين فقط بأعداد غير منطقية.
- السبب: الفكّ الخطي على قسم `.text` المُعتَّم (كود ذاتي التعديل + بيانات مضمّنة) يُنتج تعليمات وهمية، وأرقام النداء (`x8`) **مُحتسبة ديناميكيًا** لا كثوابت.
- **الخلاصة: هذا الرقم أثر خادع ولا يُعتمد.** المصدر الموثوق للنداءات هو المحاكاة (القسم 3).

## 3. 🟢 نتائج المحاكاة الفعلية (Qiling)

**المنهجية:** تحميل `libengine.so`، تهيئة كتلة TLS (الكود يقرأ حارس المكدّس من `tpidr_el0 + 0x28`)، تخطيط الصفحات عند الطلب، ومعالج نداءات نظام عام لكل الأرقام (0–520) يسجّل رقم `x8` الحقيقي وقت التنفيذ، ثم تشغيل كل الـ **44 مُنشئًا (constructors)** من `.init_array`.

### النتيجة القاطعة
```
النداءات الحقيقية المرصودة:  mmap  = 37 مرة، وكلها prot=0x7 (RWX: قراءة+كتابة+تنفيذ)
أحداث تخصيص ذاكرة قابلة للتنفيذ (PROT_EXEC): 37
```

- **كل مُنشئ** يستدعي مبكّرًا `mmap` بحماية **RWX (0x7)** — أي يخصّص ذاكرة **قابلة للتنفيذ والكتابة معًا**.
- هذا هو التوقيع الكلاسيكي لـ **توليد الكود وقت التشغيل / الكود ذاتي التعديل** (كتابة أكواب/Trampolines صغيرة ديناميكيًا).

### 🟢 التقاطع (Cross-Validation) — أقوى دليل
النتيجة الديناميكية تُطابق تمامًا الفكّ الساكن لـ `JNI_OnLoad` في Ghidra:
```c
// Ghidra (ساكن):  mmap عبر  FUN_0091ad58(addr, 0, len, 7, 0x22, -1, 0)
//                 prot=7 (RWX)   flags=0x22 (MAP_PRIVATE|MAP_ANONYMOUS)
// ثم:  *(uint*)pcVar11 = ... | 0x14000000;   // كتابة تعليمة فرع ARM64 (B) في الذاكرة المخصّصة
```
```
Qiling (ديناميكي):  37 × mmap  prot=0x7  ← نفس القيمة بالضبط
```
> **الاستنتاج (🟢 مُثبَت لا مُستنتَج):** المكتبة، عند التهيئة، تخصّص ذاكرة RWX وتكتب فيها تعليمات ARM64 مُولّدة ديناميكيًا. هذا يرفع «توليد الكود وقت التشغيل» من *استنتاج* إلى *حقيقة مرصودة ومؤكَّدة من مصدرين* (ساكن Ghidra + ديناميكي Qiling).

الملفات: `ghidra/../emulation_trace.txt`، السكربت `tools/emulate_engine.py`.

## 4. 🟢 سلوك التشغيل من طبقة Java (أدلة قاطعة من الكود المفكوك)

هذه ليست استنتاجات — إنها منطق مقروء مباشرة من كود Java:

| السلوك | الدليل (الملف) | التصنيف |
|--------|----------------|---------|
| تحميل مكتبة أصلية عند الإقلاع | `App.java`: `System.loadLibrary("engine")` في كتلة static | 🟢 |
| تحميل كود ديناميكيًا (DEX) | فئات ترث `PathClassLoader` (`lk1.java`, `fh.java`) | 🟢 |
| **تزييف فئات الانعكاس الأساسية** | `fh.loadClass`: يُرجع `Executable→cz.c`، `MethodHandle→cz.e`، `Class→cz.b` | 🟢 |
| **Hooking لدوال Java** | `Native.update(Object, Method)` + `MethodUtils` (بناء توصيفات JNI) | 🟢 |
| **إعادة تعيين UID/PID** | `Native.gcuid(int)` → `yu0.u().r(Binder.getCallingPid())` | 🟢 |
| فك تشفير سلاسل/بيانات وقت التشغيل | `Native.ilil(int)→String`، `djp(int)→byte[]` | 🟡 (الأسماء تؤكّد الوظيفة) |
| فحص ترخيص/توقيع | `Native.chl(byte[])→boolean` | 🟡 |
| خدمة دائمة (Persistence) | `DaemonService`: تُعيد جدولة نفسها عبر `AlarmManager` عند القتل | 🟢 |
| IPC للحاوية الافتراضية | `SystemCallProvider.call("VM", …)` + `_S_|_server_` | 🟢 |
| الوصول لسياقات حزم أخرى | `createPackageContext` (11 موضعًا) | 🟢 |

### التوصيف المعماري (🟡 دليل قوي)
مجموع الأدلة يشير بقوة إلى إطار **محاكاة تطبيقات (App-Virtualization / VirtualApp-style)** مدمج مع **محرّك Hooking أصلي (Inline Hook)**: يشغّل تطبيقات/ألعابًا مستهدفة داخل حاويته، يزيّف الهوية (UID/PID + فئات الانعكاس)، ويعدّل سلوك الدوال عبر أكواب مُولّدة في ذاكرة RWX.

> ملاحظة دقّة: هذا **توصيف وظيفي مدعوم بأدلة قوية**، وليس نسبةً لمشروع بعينه. الجزم بأداة/مشروع محدّد يتطلّب مطابقة بصمات إضافية.

## 5. 🟡 حول «نسبة الحماية إلى OLLVM»

السمات المرصودة (تسطيح تدفّق التحكّم، دوال ضخمة مُولّدة، تشفير سلاسل، أكواب RWX) **متوافقة مع OLLVM ومشتقّاته**، لكنها **ليست حصرية له**. لذا تُعامَل كـ«تعتيم شبيه بـ OLLVM (OLLVM-style)» — 🟡 دليل قوي وليس جزمًا. لا توجد بصمة أداة بناء صريحة تُثبت المُنتِج بعينه.

---

## 6. ماذا يلزم لإكمال ما تعذّر ساكنًا/بالمحاكاة الجزئية؟

للوصول للسلوك الكامل (فك أسماء JNI الفعلية، الملفات المُنشأة، حركة الشبكة، تحميل DEX الهدف):
1. **جهاز/محاكي أندرويد arm64 حقيقي + Frida** (يتطلّب KVM غير متوفّر هنا):
   - `Interceptor.attach` على `RegisterNatives` لاستخراج (اسم الدالة ↔ المؤشّر).
   - Hook على `mmap`/`mprotect`/`dlopen`/`openat` لرصد كل تحميل/تعديل فعلي.
   - `blutter_frida.js` (مُرفق) لتتبّع دوال Dart.
2. **تفريغ الذاكرة بعد التهيئة** لالتقاط الكود بعد فك تعديله ذاتيًا (تجاوز التعتيم).
3. متابعة المحاكاة بـ Qiling مع بناء بيئة JNI/bionic أكمل (جهد بحثي أكبر).
